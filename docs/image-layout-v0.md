# raisOS Image And Persistence Layout V0

Status: draft specification, no kernel write path implemented.

This document defines the first durable image/persistence layout raisOS should
target before Stage-0 writes anything to disk. The current boot image remains a
single FAT32 boot image; the V0 layout below is the next architecture boundary,
not an assertion that the current kernel already supports it.

## Goals

- Preserve the current bootable Stage-0 path while making later persistence
  explicit and recoverable.
- Keep boot selection understandable by firmware, Limine, and a very small
  raisOS core.
- Support rollback with `pending`, `success`, and `last_good` state before any
  service or kernel update is allowed to become permanent.
- Provide a DATA partition for persistent state that is separate from bootable
  ESP contents.
- Fail closed: missing or corrupt evidence must boot the last known-good system
  or SAFE mode, not silently accept a candidate.
- Avoid provider keys in default images or tracked staging trees.

## Current Stage-0 Image

The verified MVP image is:

```text
release/raisos-stage0.img
```

It is currently generated by `scripts/package-stage0.ps1`, which builds the
kernel and calls `scripts/make-fat32-image.py` with a fixed size of 67108864
bytes. The image is a single FAT32 filesystem, not a GPT disk with multiple
partitions.

Current files written into the image are:

```text
/limine.conf
/EFI/BOOT/BOOTX64.EFI
/EFI/BOOT/limine.conf
/kernel/kernel.elf
```

The tracked staging tree also contains `release/esp/kernel/seed-kernel.elf` and
`release/esp/limine-uefi-cd.bin`, but the Windows FAT32 image builder only
copies the files listed above.

Current runtime properties:

- Limine 10 loads `/kernel/kernel.elf` from the same FAT32 filesystem.
- Stage-0 treats the image as read-only at runtime.
- Provider API keys are RAM-only unless a separate ignored local image is built
  with `-UseTempEsp -EmbedOpenAiApiKeyFromEnv`.
- There is no DATA partition, boot-state file, A/B slot, journal, safe-mode
  marker, or kernel persistence path.

The current single-FAT image is therefore an MVP boot carrier only. It must not
grow ad-hoc runtime writes.

## V0 Disk Layout

V0 should use GPT once persistence exists. Partition names are part of the
contract so the core can report and validate them through future snapshot and
agent protocol messages.

```text
Partition  Name           Type                  Role
---------  -------------  --------------------  ------------------------------
1          SEED_ESP_A     EFI System Partition  Primary UEFI/Limine boot slot
2          SEED_ESP_B     EFI System Partition  Alternate UEFI/Limine boot slot
3          SEED_DATA      raisOS DATA           Persistent state and boot facts
4 optional SEED_RECOVERY  EFI or raisOS image   Factory/recovery boot assets
```

Initial recommended sizes:

```text
SEED_ESP_A     128 MiB FAT32
SEED_ESP_B     128 MiB FAT32
SEED_DATA      remaining space, minimum 256 MiB for development images
SEED_RECOVERY  optional 128 MiB, omitted until recovery assets exist
```

`SEED_ESP_A` and `SEED_ESP_B` contain complete bootable slot payloads:

```text
/EFI/BOOT/BOOTX64.EFI
/EFI/BOOT/limine.conf
/limine.conf
/kernel/kernel.elf
/raisos/slot.json
/raisos/manifest.json
```

`SEED_DATA` contains mutable state only:

```text
/raisos/boot/control.json
/raisos/boot/control.json.tmp
/raisos/boot/control.json.prev
/raisos/boot/events.log
/raisos/state/
/raisos/services/
/raisos/artifacts/
/raisos/audit/
```

The ESP slots are boot payloads. DATA is the only normal persistence target.
Stage-0 must not store provider keys, chat logs, module state, or rollback
control data in the ESP slots.

## ESP-A/B Versus Kernel-A/B

V0 chooses ESP-A/B as the first concrete layout.

Reasons:

- UEFI and Limine already understand FAT32 ESP content.
- Each slot can be inspected by mounting a normal FAT filesystem.
- Bootloader, Limine config, and kernel stay together, which avoids a slot where
  Limine and kernel disagree about paths or protocol expectations.
- The current single-FAT image can evolve into `SEED_ESP_A` without changing the
  boot file structure.

Kernel-A/B inside one ESP is a possible later simplification for tiny media:

```text
/kernel/a/kernel.elf
/kernel/b/kernel.elf
/raisos/boot/control.json
```

It is not the V0 default because a corrupt or incompatible shared ESP can break
both kernel slots at once. If Kernel-A/B is later added, it must preserve the
same `pending`, `success`, `last_good`, and SAFE mode state model.

## Slot Metadata

Each ESP slot should include `/raisos/slot.json`:

```json
{
  "schema": "raisos.slot.v0",
  "slot": "A",
  "generation": 12,
  "kernel_path": "/kernel/kernel.elf",
  "kernel_sha256": "hex",
  "limine_config_sha256": "hex",
  "created_utc": "2026-05-17T00:00:00Z",
  "requires_data_schema": "raisos.data.v0"
}
```

Each ESP slot should also include `/raisos/manifest.json` with hashes for all
slot files. The manifest is read-only boot evidence; it is not a trust root by
itself. Signature and attestation requirements belong to later module and
update gates.

## Boot Control State

`SEED_DATA:/raisos/boot/control.json` is the authoritative mutable boot-control
record:

```json
{
  "schema": "raisos.boot_control.v0",
  "active": "A",
  "last_good": "A",
  "pending": null,
  "safe_mode": false,
  "boot_attempt": {
    "slot": "A",
    "generation": 12,
    "attempt_id": "uuid-or-monotonic-id",
    "started_utc": "2026-05-17T00:00:00Z",
    "success_marked": true
  },
  "slots": {
    "A": {
      "generation": 12,
      "state": "good",
      "failure_count": 0,
      "last_success_utc": "2026-05-17T00:00:00Z"
    },
    "B": {
      "generation": 13,
      "state": "empty",
      "failure_count": 0,
      "last_success_utc": null
    }
  }
}
```

Slot states:

```text
empty       slot has no complete manifest
candidate   slot was written but has not been selected
pending     slot is selected for a probation boot
good        slot completed boot-success criteria
bad         slot failed validation or exceeded boot attempts
```

Rules:

- `pending` names a candidate slot and generation that may be booted once or a
  bounded number of times.
- `success_marked=true` means the booted kernel reached the explicit success
  condition and atomically recorded it.
- `last_good` is updated only after success is marked.
- If `pending` exists but success was not marked for its attempt, the next boot
  must prefer `last_good`.
- If DATA is missing or unreadable, Stage-0 may boot the current ESP payload but
  must report persistence unavailable and must not mark anything good.

## Boot Flow

The intended V0 flow is:

1. Firmware loads the default UEFI path from the selected ESP.
2. Limine loads the slot kernel from the same ESP.
3. The raisOS core mounts DATA read-only first.
4. The core reads `control.json` and validates the active slot metadata.
5. If the current slot is pending, the system enters probation.
6. Non-core persistent writes remain denied during early boot.
7. After required health checks pass, the core writes a boot-success mark.
8. Only after success may `last_good` move to the new slot.

The current Stage-0 kernel does not implement these steps yet. The V0 rule is
that writes become available only after the read/validate/report path exists.

## Boot Success Criteria

V0 boot success should be narrow and machine-checkable:

```text
framebuffer initialized or serial fallback active
core memory allocator initialized
input status reported, even if degraded
DATA mounted and boot-control state readable/writable
service inventory/snapshot root available
no core panic before success timeout
```

Network, provider login, Wi-Fi association, and AI response success are not V0
boot-success requirements. They are services and may be degraded without making
the slot unbootable.

## SAFE Mode

SAFE mode is a boot mode controlled by `control.json`, a physical/key command
path later, or repeated failure policy.

SAFE mode properties:

- boot only permanent core and minimal diagnostics
- disable non-core modules and live-loaded services
- disable provider requests by default
- disable all persistent writes except SAFE-mode audit and explicit recovery
  writes
- do not consume or advance `pending`
- expose clear status: `SAFE MODE`, selected slot, `last_good`, and DATA state

SAFE mode is not a fallback that pretends the candidate worked. It is an
explicit degraded mode for inspection and recovery.

## Atomic Writes

All V0 DATA writes must be replace-style writes with verification:

1. Write the new content to a temporary file in the same directory.
2. Flush file data.
3. Re-read and verify length, schema, and hash.
4. Rename current file to `.prev` where needed.
5. Rename temporary file to the final name.
6. Flush directory metadata if the filesystem supports it.
7. Append a compact audit event after the state transition.

For FAT32, rename ordering and power-failure behavior are weaker than on a
journaled filesystem. V0 must therefore keep `control.json.prev` and validate
both files on boot. If both records are invalid or contradictory, boot selection
must fall back to SAFE mode or the currently loaded slot with persistence
disabled.

No partial binary patching of kernel images is allowed in V0. A candidate slot
is written as a complete slot, verified by manifest hashes, then selected by a
small boot-control update.

## Persistence Boundaries

Allowed in `SEED_DATA` after V0 support exists:

- boot-control records
- boot and recovery audit events
- service inventory snapshots
- module manifests and candidate artifacts after validation
- non-secret user settings
- crash records and test reports

Denied in default persistent storage until a stronger secret design exists:

- OpenAI or provider API keys
- Wi-Fi passphrases
- bearer tokens, cookies, or pairing secrets
- unredacted provider prompts containing local-only or secret fields

Secrets need a separate design with hardware binding, encryption, or an explicit
local unlock flow. Until then, the current RAM-only provider key behavior remains
the correct default.

## What Stage-0 Does Not Do Yet

Stage-0 currently does not:

- parse GPT
- select between ESP-A and ESP-B
- mount a DATA partition
- write `control.json`
- maintain `pending`, `success`, or `last_good`
- boot SAFE mode
- atomically write persistent state
- verify slot manifests
- persist service or provider configuration
- recover from a failed persistent update
- protect secrets at rest

Any implementation work must expose these as explicit missing capabilities until
the real read/validate/write path is present.

## Open Questions

- Should V0 DATA use FAT32 for early simplicity or a small log-structured raisOS
  filesystem for stronger crash behavior?
- Should boot selection be driven by UEFI boot variables, duplicated removable
  media paths, or a tiny stable boot manager in each ESP?
- What exact success timeout and failure-count threshold should move a pending
  slot back to `last_good`?
- Which hardware-backed secret mechanism is acceptable for provider keys on
  machines without TPM support?
- How should `vm_test_report.v0` and local attestation bind to a slot
  generation before persistence is allowed?
